# 15 — n8n Workflow Automation

[n8n](https://n8n.io) — open-source workflow automation platform. Connects services via visual flows (webhooks, schedules, API calls, etc.).

## Stack

| Container | Image | Purpose |
|-----------|-------|---------|
| `n8n` | `n8nio/n8n:latest` | Automation engine + web UI |
| `n8n-postgres` | `postgres:16-alpine` | PostgreSQL database |

Two containers on a shared user-defined Docker network (`n8n_default`) so n8n can
reach Postgres by hostname. Migrated from SQLite → Postgres on 2026-05-29 (see
[`journal/2026-05-29.md`](../journal/2026-05-29.md)) for a more robust backend
ahead of heavier scheduled workflows.

## Install

[`compose/n8n.yml`](../compose/n8n.yml).

```bash
mkdir -p /mnt/drive1/AppData/n8n
chown -R 1000:1000 /mnt/drive1/AppData/n8n
```

The `chown` is required — n8n runs as UID 1000 inside the container and needs write access on first boot to generate its encryption key.

```bash
docker compose -f /mnt/drive1/AppData/compose/n8n/docker-compose.yml up -d
```

## Configuration

Edit the env vars in the compose file before starting:

| Variable | What to set |
|----------|-------------|
| `N8N_HOST` | Your subdomain, e.g. `n8n.yourdomain.com` |
| `WEBHOOK_URL` | Full URL with trailing slash, e.g. `https://n8n.yourdomain.com/` |
| `GENERIC_TIMEZONE` | Your TZ, e.g. `America/New_York` |
| `N8N_ENCRYPTION_KEY` | Copy the value from `/mnt/drive1/AppData/n8n/config`. Pinning it in the compose makes it explicit and survives a config-file loss. |
| `POSTGRES_PASSWORD` / `DB_POSTGRESDB_PASSWORD` | Same strong value in both services (`openssl rand -hex 24`). |

## Data location

```
/mnt/drive1/AppData/n8n/
├── config           # encryption key (auto-generated on first boot; also pinned in compose)
├── nodes/           # community nodes
└── storage/         # binary data from executions

/mnt/drive1/AppData/n8n-postgres/   # PostgreSQL data directory (workflows, credentials, executions)
```

> Workflows/credentials/executions now live in **Postgres** (`n8n-postgres`),
> not in `database.sqlite`. The old SQLite file and a pre-migration backup are
> retained under `/mnt/drive1/AppData/n8n/_migration_backup_*/`.

## Workflows

### New Media → Notification

Sends a Telegram message when Sonarr/Radarr finish importing. Sanitized export:
[`configs/n8n-workflow-new-media-notification.example.json`](../configs/n8n-workflow-new-media-notification.example.json)
(import via n8n → *Workflows → Import from File*).

Flow: **Webhook** (`POST /webhook/media-added`) → **Code** (detects Sonarr vs
Radarr payload, builds the message) → **Telegram** (sends to a chat).

Setup:
1. Telegram: message `@BotFather` → `/newbot` → get a **bot token**. Message your
   new bot once, then read your **chat ID** from
   `https://api.telegram.org/bot<TOKEN>/getUpdates`.
2. In n8n, create a **Telegram API** credential with the bot token; set the chat
   ID on the Telegram node. (Token/chat ID are intentionally *not* in the repo —
   placeholders `CHANGE_ME` are.)
3. Activate the workflow.
4. In **Sonarr** and **Radarr**: *Settings → Connect → + → Webhook*
   - URL: `http://192.168.50.178:5678/webhook/media-added` (LAN IP — *arr and n8n
     are on different Docker networks, so don't use a container hostname; the
     public `https://n8n.asj.media/webhook/media-added` also works)
   - Method: `POST`; triggers: **On Import / On Download** (and On Upgrade if wanted)
   - Hit **Test** — you should get a Telegram message.

All workflows below reuse the **same** Telegram credential / bot.

### Torrent Done → Telegram

Notifies on **every** finished qBittorrent torrent (incl. manual grabs the *arr
stack never imports), and flags uncategorized ones.
Export: [`configs/n8n-workflow-torrent-done.example.json`](../configs/n8n-workflow-torrent-done.example.json)

Flow: **Webhook** (`POST /webhook/torrent-done`) → **Code** (format) → **Telegram**.

Hook it up in qBittorrent → *Options → Downloads → Run external program on torrent
completion* (qBit's container has `curl`):

```sh
curl -s -X POST http://192.168.50.178:5678/webhook/torrent-done \
  --data-urlencode "name=%N" --data-urlencode "category=%L" \
  --data-urlencode "size=%Z" --data-urlencode "files=%C" \
  --data-urlencode "content=%F" --data-urlencode "savepath=%D"
```

### Disk Guard → Telegram

Alerts when any `/mnt` data drive crosses **85%** full, and again when it
recovers (de-duped — one alert per state change, no spam).
Export: [`configs/n8n-workflow-disk-guard.example.json`](../configs/n8n-workflow-disk-guard.example.json)

Flow: **Webhook** (`POST /webhook/disk-report`) → **Code** (per-drive threshold +
de-dup via workflow static data) → **Telegram**.

Because the n8n image is hardened (no Execute Command node, Code-node sandbox
locked down), disk stats are gathered **on the host** and pushed in:

- Script: [`scripts/n8n-disk-report.sh`](../scripts/n8n-disk-report.sh) → installed
  at `/usr/local/bin/n8n-disk-report.sh` (read-only `df` only)
- Cron (root): `*/30 * * * * /usr/local/bin/n8n-disk-report.sh`

This monitors **all** drives (drive1–4 + toshiba), not just the download target.

### Soulseek (slskd) Done → Telegram

Polls slskd every 5 min and notifies (grouped by album) when downloads finish.
Seeds history on first run so it doesn't blast old downloads.
Export: [`configs/n8n-workflow-soulseek-done.example.json`](../configs/n8n-workflow-soulseek-done.example.json)

Flow: **Schedule (5 min)** → **HTTP Request** (`GET /api/v0/transfers/downloads`)
→ **Code** (new-completed detection via static data) → **Telegram**.

slskd has no completion hook, so this uses its API — which needs an **API key**.
In `slskd.yml` under `web.authentication`:

```yaml
    api_keys:
      n8n:
        key: CHANGE_ME_SLSKD_API_KEY        # openssl rand -hex 24
        cidr: 192.168.0.0/16,172.16.0.0/12,10.0.0.0/8,127.0.0.1/32  # private nets only
```

Restart slskd, then store the key in n8n as an **HTTP Header Auth** credential
(`X-API-Key`). (Music Assistant usually auto-detects new files; a forced rescan
via its websocket API can be added later if needed.)

### Todo Capture → Todoist (via Gemini)

Text the **ASJ Noti** bot a rough note; Gemini (`gemini-3.5-flash`) cleans it into
a proper task and creates it in **Todoist** (title, natural-language due date,
priority, category as a label). Bot replies with a ✅ confirmation.
Export: [`configs/n8n-workflow-todo-capture-todoist.example.json`](../configs/n8n-workflow-todo-capture-todoist.example.json)

Flow: **Schedule (1 min)** → **Read offset** (Code) → **HTTP** (`getUpdates?offset=&limit=1`)
→ **Get message** (Code) → **Gemini** (HTTP, retry on fail) → **Parse task** (Code)
→ **Create Todoist task** (HTTP `POST /api/v1/tasks`, retry) → **Confirm** (Telegram)
→ **Mark read** (Code).

Key design points (learned the hard way):

- **Outbound polling, not a Telegram Trigger.** n8n.asj.media is LAN-only, so
  Telegram's servers can't reach an inbound webhook. The workflow *polls*
  `getUpdates` (outbound) instead — no public exposure.
- **Mark-read-only-after-success.** The Telegram offset is advanced in the final
  **Mark read** node, *after* the Todoist task is created — never before. A
  failure just leaves the message queued for the next poll, so messages are
  never lost and never duplicated. (Earlier design advanced the offset up front
  and silently dropped messages when Gemini got rate-limited.)
- **One message per poll** (`limit=1`) avoids bursting Gemini's free tier (429s).
- **No hardcoded project IDs.** Tasks go to Inbox; the category is applied as a
  **label** (auto-creates, never 404s when you reorganize Todoist).
- Credentials in n8n: Telegram (bot token), **Gemini API key** (HTTP Header
  `x-goog-api-key`, free tier from aistudio.google.com), **Todoist API**
  (HTTP Header `Authorization: Bearer …`).

> Gemini note: the free-tier key uses `gemini-3.5-flash`. The consumer Gemini
> *subscription* can't be used by automations — only an **API key** from Google
> AI Studio works.

### Things → Todoist Import (one-time migration)

Migrated an existing **Things 3** task list to Todoist. Things has no cloud API,
so an **Apple Shortcut** on iPhone reads open to-dos (Find Items → Dictionary of
title/notes/when/area → POST) to this webhook, which creates them in Todoist.
Export: [`configs/n8n-workflow-things-import.example.json`](../configs/n8n-workflow-things-import.example.json)

Flow: **Webhook** (`POST /webhook/things-import`) → **Code** (parse, dedupe vs
existing, create each via Todoist API, preserve Things Area/Project as a
`Things_*` label).

- **Duplicate-safe** (skips tasks whose title already exists) and **date-tolerant**
  (drops a malformed date rather than failing the whole task).
- **Things "Mail to Things" only carries title+notes** — no dates/projects — which
  is why the URL-scheme/Shortcut route was needed for structure.
- Free Todoist caps at 5 projects, so structure is preserved as **labels**, not
  projects.

### FlacPlayer TODO.md → Todoist (one-way mirror)

Mirrors open items from the **private** `AndresASJ/FlacPlayer` repo's `TODO.md`
into Todoist under a **`FlacPlayer`** label. The file is the **source of truth**;
new `- [ ]` items appear in Todoist, and items checked off / removed in the file
get completed in Todoist. The repo is read-only — the doc is never written back
(it's a 600-line tiered design doc; two-way sync would mangle it).

Not an n8n workflow — a host script + cron (reuses the box's existing `gh` auth,
so no GitHub token is needed in n8n):

- Script: [`scripts/flacplayer-todo-sync.py`](../scripts/flacplayer-todo-sync.py)
  → installed at `/usr/local/bin/flacplayer-todo-sync.py`
- Todoist token read from `/root/.config/flac-sync/todoist.token` (chmod 600;
  **not** in the repo)
- Cron (root): `7 * * * *` (hourly)

Two Todoist quirks handled (learned the hard way):
- **Todoist strips `@word`/`#word` from task content** (it reads them as
  label/project syntax) — so titles are **backslash-escaped** (`\@Query`) to
  display intact and avoid junk labels.
- Because of that mangling, matching is **never by title**. Each task carries a
  stable `fpid:<hash>` marker in its **description** (descriptions aren't parsed),
  making the sync idempotent (re-running is a clean no-op).

## Notes

- The "Python task runner" warning on startup is harmless — Python isn't installed in the image, but JavaScript/Node runners still work.
- `N8N_SECURE_COOKIE=false` is set so the HTTP connection from NPM to n8n doesn't cause cookie auth failures.
- **Backups:** include both `/mnt/drive1/AppData/n8n/` (encryption key + binary data) **and** `/mnt/drive1/AppData/n8n-postgres/` in your backup script. For consistent DB dumps, prefer `docker exec n8n-postgres pg_dump -U n8n n8n > n8n.sql`.
- n8n does **not** auto-migrate data between SQLite and Postgres — switching `DB_TYPE` starts from an empty schema. The migration on 2026-05-29 was clean because the SQLite DB held no workflows/credentials (only the initial owner account, which had to be re-created once on Postgres).
