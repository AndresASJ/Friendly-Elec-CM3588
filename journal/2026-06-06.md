## Hermes: read the chat logs, root-caused the "can't do anything" failures

Reviewed the actual Telegram conversations from Hermes' session store (`/opt/data/state.db`,
151 messages / 3 sessions). One root cause behind most of it, plus several behavior fixes.

### Root cause: Hermes couldn't read its own homelab memory

`/opt/data/memories/USER.md` (the Radarr/Sonarr/Lidarr/n8n playbook) had been left
**`root:root` mode 600** after a root edit on Jun 3 16:31 — but the runtime user is **uid
10000**, so it couldn't open the file. Since that restart, every session ran with **zero**
homelab knowledge. That's why the Jun 5 "download Man on the Run" request degraded into
web-scraping DuckDuckGo (it even solved a CAPTCHA), got 403'd, then refused on "copyright"
grounds — and why "go through sonarr/radarr" → "they're not installed." Same `edited-as-root`
trap already documented for `config.yaml`.

- **Fixed:** `chown 10000:10000 USER.md && chmod 640` (match `config.yaml`). Verified readable
  as uid 10000; content intact. No restart needed — USER.md is re-read at session start.

### Request routing — teach it to triage a bare request

USER.md documented *how* to drive each service but nothing told the agent *which* service a
bare request maps to. Added a **"Request routing — READ THIS FIRST"** section at the top of
USER.md: movie/film/documentary → Radarr, TV/show → Sonarr, album/song/artist → Lidarr,
to-do → Todoist. Explicit: **never scrape search engines, solve CAPTCHAs, or refuse media on
copyright grounds** — it all runs through the owner's stack. For a bare title of unknown type
(the "Man on the Run" case), query *both* Radarr `/movie/lookup` and Sonarr `/series/lookup`
and confirm the match. URL-encode lookup terms (`%20`) — a raw space made `curl` exit 3.
Confirmed working: re-asking now hits Radarr instead of the open web.

### Persona — wrote a real SOUL.md (was the empty default)

`/opt/data/SOUL.md` was still the shipped template, so nothing steered tone or behavior — hence
the verbose, apologetic, 3-option-menu replies in the logs. Wrote a concise persona: direct
voice, **bias to action** (act on clear requests, confirm only the matched title before adding),
route-through-the-stack reflex, and **report outcomes honestly** (no fake success). Loaded fresh
every message — live immediately.

### Self-learning memory — gave the agent a writable store

The memory tool had been **refusing to auto-update USER.md** ("content wouldn't round-trip").
Cause: tool-managed memory is `§`-delimited entries under tight char limits (USER.md 1375,
MEMORY.md 2200); our ~9 KB hand-authored playbook is neither. It loads fine as a system-prompt
snapshot (so routing works) but can't be tool-written. Fix matching the two-file model
(`tools/memory_tool.py`): keep **USER.md** as the human-authored playbook, and seeded an empty
**MEMORY.md** as the agent's writable self-curated store. A note atop USER.md tells the agent to
record learnings in MEMORY.md and never tool-edit USER.md. Self-curation loop works again.

### Todoist — fixed the lie, the scraped token, and due dates

In the logs Hermes had **no Todoist token in its env**, so it scraped the **hardcoded** token
out of the n8n "Things → Todoist Import" workflow JS (also leaking it into chat), used the wrong
webhook, and **falsely reported** "rescheduled to Sunday 3pm" when n8n actually returned
`created:0, skipped:1` (deduped, due date silently dropped).

- Added `TODOIST_API_TOKEN` to `/opt/data/.env` (from the canonical
  `/root/.config/flac-sync/todoist.token`; validated read-only `GET /tasks` → 200). Restarted
  Hermes via `compose down && up -d` (per the s6-lock gotcha — not bare `restart`); clean boot,
  Telegram reconnected. The token loads into the agent's shell the same way `LIDARR_API_KEY`
  does (`load_dotenv` → `os.environ`).
- Added a **Todoist section** to USER.md: drive the v1 API directly with
  `Authorization: Bearer $TODOIST_API_TOKEN` (read from env, never hardcode/echo); set due dates
  via natural-language `due_string`; reschedule via `POST /tasks/{id}`; file into sections
  (Kord/Blog/Church/Coding/Routines); and **report the response literally** — only claim
  success when the API returns the task. Don't touch FlacPlayer/automation-owned tasks.

**Token rotated (later same day):** owner issued a new Todoist token (Settings → Integrations →
Developer → "Issue a new API token"). Wrote a turnkey rotation script
(`/root/.config/flac-sync/rotate-todoist-token.py`) that validates the new token, backs up and
updates all three locations (flac-sync token file, Hermes `.env`, n8n workflow `EsEMth8aREpDrYcn`
node "Import to Todoist", swapped via the n8n REST API), restarts Hermes, and verifies the result.
Ran clean: **new token → 200, old token → 401** everywhere; Hermes rebooted (Telegram connected),
workflow still active. The leaked token is now dead.

**Residual (optional hardening):** the new token is still a *literal* in the n8n Code node. Moving
it to an n8n env var/credential (so it's not in the workflow JSON at all) needs an n8n container
recreate — deferred unless wanted.
