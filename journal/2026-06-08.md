## Voice/VA: Follow-Up Mode — chain commands without repeating the wake word

Owner wanted it to feel like Alexa/Google ("don't make me say the wake word every time").
Clarified the real model: Alexa/Google aren't wake-word-free either — they use **Follow-Up
Mode** / **Continued Conversation** (wake once, then a short open-mic window for the next
command). True always-on STT is a bad idea even when local: TV/music/chatter constantly
mis-fire. So the target is a follow-up window, not open mic.

HA's built-in `continue_conversation` is hardcoded (`conversation/chat_log.py`) to only fire
when the assistant's reply **ends in "?"** — i.e. clarifications continue, plain confirmations
("Turned on the light.") don't. No native "always follow-up" toggle.

Built a real Follow-Up Mode using `assist_satellite.start_conversation`, which (verified in
`services.yaml`) accepts an **empty `start_message`** + **`preannounce: false`** → it
**silently re-opens the mic** (no chime, no spoken prompt). Automation
`voice_follow_up_mode` (`automations.yaml`): trigger = satellite
`assist_satellite.vaca_de0acdae6` state `responding` → `idle`; action = silent
`start_conversation` re-arm. Works for **all** commands incl. the local fast-path (not just
Gemini turns).

**Self-terminating / no runaway loop:** when no command is spoken, the re-armed mic times out
from `listening` → `idle` (never `responding`), so the trigger doesn't re-fire — the chain
ends on its own. Confirmed via 36h history that the satellite genuinely cycles
idle→listening→processing→responding→idle, so the trigger is reliable.

Applied without a restart: backed up `automations.yaml`
(`_ha-config-backups/automations.yaml.bak-*`), edited, `automation.reload` via API (HTTP 200).
Verified `automation.voice_follow_up_mode_no_wake_word_for_next_command` = **on**.
**Fully reversible:** toggle the automation off in Settings → Automations to disable Follow-Up
Mode. Owner to test on the Show: "Alexa, turn on the lights" → (mic silently reopens) → "turn
off the fan" → … → stop talking and the window closes.

## Jellyfin/Jellyseerr: added user "Hendrix" (friend)

Created a Jellyfin account for a friend (Hendrix) and gave him media-request access via
Jellyseerr.

- **Jellyfin user `Hendrix`** created via API (`POST /Users/New`, used the existing admin API
  key that Jellyseerr already holds — no new key minted). Default policy left in place: **not**
  admin, access to both libraries (Movies + Shows), playback + downloading allowed, content
  deletion disabled. Password set and shared with the owner out-of-band (not stored in repo).
- **Jellyseerr import** via `POST /api/v1/user/import-from-jellyfin` so Hendrix can sign in to
  Jellyseerr (port 5055) with his Jellyfin credentials and submit requests. Default Jellyseerr
  user permissions (request, no auto-approve/admin).
- Reversible: delete the user in Jellyfin Dashboard → Users, and remove him in Jellyseerr →
  Users.
