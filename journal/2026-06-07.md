## Voice/VA: response text lingered 2 min on the Echo Show ‚Äî cut to 10s

Owner noticed the Show displays the spoken response, then leaves it on screen far too
long before returning to the clock. Not a bug ‚Äî a View Assist setting.

Root cause: View Assist's **Master Configuration** entry
(`01KJJMDK71WAJ6YPGZH1TET9C7`, domain `view_assist`, `type: master_config`) had
`view_timeout: 120.0`. That's the master fallback for how long any non-default view ‚Äî
including the **intent view** that shows the spoken reply ‚Äî holds before reverting to the
home/clock view. The Echo Show 8 device entry (`01KJJR748KBM4BX6MPR22HXX68`) has empty
options, so it inherits the master value ‚Üí every reply lingered a full 2 minutes.

Fix: set `view_timeout` ‚Üí **10.0** in the master config options. Applied via the safe
HA-config path (back up `core.config_entries` ‚Üí `docker stop homeassistant` ‚Üí JSON edit ‚Üí
`docker start`), since options/subentry flows aren't reliably exposed over the API on this
HA version. Backup: `_ha-config-backups/core.config_entries.bak-1780852621`. HA came back
HTTP 200 in ~10s; verified live store reads `[10.0]`; View Assist loaded clean (only the
standard "untested custom integration" warning). View Assist reads this live ‚Äî no device
reboot needed; applies from the next interaction.

Tunable later from the HA UI: *Settings ‚Üí Devices & Services ‚Üí View Assist ‚Üí Master
Configuration ‚Üí Configure ‚Üí Display settings ‚Üí View timeout*. (Related knobs in the same
entry: `menu_timeout: 10`, `music_mode_timeout: 600`.)

## Voice/VA: faster device commands + British voice

Two owner-requested tweaks to the **Local Voice (Whisper+Gemini+Piper)** Assist pipeline.

**1. Faster replies ‚Äî handle commands locally first.** The pipeline had
`prefer_local_intents: False`, so *every* utterance (even "turn on the lights") took the
Gemini cloud round-trip. Set it **True**: HA now matches its built-in local intent engine
first for exposed entities (lights/scenes/etc.) and only falls through to Gemini for
fuzzy/Q&A requests. Device commands resolve on-box, near-instant, and cut Gemini calls.
(Note: `prefer_local_intents` is a **pipeline** field in `assist_pipeline.pipelines`, not a
Google Generative AI integration option ‚Äî the integration's const has no local-intent key.)

**2. British female voice.** Swapped Piper TTS from `en_US-lessac-medium` to
**`en_GB-jenny_dioco-medium`** (clean RP British female, medium quality ‚Äî fast enough on the
RK3588; `en_GB-cori-high` is nicer but heavier). Two coordinated changes:
- `compose/wyoming-voice.yml` (and the live CasaOS `wyoming-voice` compose) ‚Üí `PIPER_VOICE`
  updated; `docker compose up -d piper` recreated the container, which downloaded the ~63 MB
  model and came back "Ready".
- The pipeline's `tts_voice` ‚Üí `en_GB-jenny_dioco-medium` to match.

Applied the pipeline edits via the safe path (back up `assist_pipeline.pipelines` ‚Üí
`docker stop homeassistant` ‚Üí JSON edit ‚Üí `docker start`; backup
`assist_pipeline.pipelines.bak-1780853323`). HA back HTTP 200; verified live store reads
`prefer_local_intents: True`, `tts_voice: en_GB-jenny_dioco-medium`; log clean. End-to-end:
re-authed to the HA API and `tts_get_url` rendered a British-English test phrase to MP3
successfully (proves the voice id is valid and Piper serves it). Old `en_US-lessac-medium`
model left on disk (harmless; can prune later).
