# 2026-06-07

## Voice/VA: response text lingered 2 min on the Echo Show — cut to 10s

Owner noticed the Show displays the spoken response, then leaves it on screen far too
long before returning to the clock. Not a bug — a View Assist setting.

Root cause: View Assist's **Master Configuration** entry
(`01KJJMDK71WAJ6YPGZH1TET9C7`, domain `view_assist`, `type: master_config`) had
`view_timeout: 120.0`. That's the master fallback for how long any non-default view —
including the **intent view** that shows the spoken reply — holds before reverting to the
home/clock view. The Echo Show 8 device entry (`01KJJR748KBM4BX6MPR22HXX68`) has empty
options, so it inherits the master value → every reply lingered a full 2 minutes.

Fix: set `view_timeout` → **10.0** in the master config options. Applied via the safe
HA-config path (back up `core.config_entries` → `docker stop homeassistant` → JSON edit →
`docker start`), since options/subentry flows aren't reliably exposed over the API on this
HA version. Backup: `_ha-config-backups/core.config_entries.bak-1780852621`. HA came back
HTTP 200 in ~10s; verified live store reads `[10.0]`; View Assist loaded clean (only the
standard "untested custom integration" warning). View Assist reads this live — no device
reboot needed; applies from the next interaction.

Tunable later from the HA UI: *Settings → Devices & Services → View Assist → Master
Configuration → Configure → Display settings → View timeout*. (Related knobs in the same
entry: `menu_timeout: 10`, `music_mode_timeout: 600`.)
