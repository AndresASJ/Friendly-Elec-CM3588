# 2026-06-01

## Rick & Morty re-grab: Remux → x265 (reclaimed ~126 GB on drive4)

Finished the R&M re-grab that was left "blocked" on 2026-05-31. Net result: the whole
series went from **163.2 GB → 37.1 GB** (drive4 library folder 35 GB), all seasons now
clean x265 1080p, **zero Remux remaining**.

### The blocker: seedpool download-slot limit (ratio standing, not concurrency)

The 6 AMZN WEB-DL packs Sonarr had grabbed from **seedpool** were all stalled at 0%,
0 seeds. Tracker message on every one: **`Download slot limit reached`**. I first
assumed it was a *concurrency* cap and tried staggering (pause all but 2, reannounce,
then down to a single torrent) — **still rejected even with ONE active torrent**. So
it's an **account-standing limit**: seedpool throttled this account's download slots to
~0 after the 3-week dead-VPN window tanked our ratio there. Staggering can't fix that;
it only recovers as ratio rebuilds (slow) — or you grab elsewhere.

Also discovered **13 junk torrents** hogging the picture: revived "with Audio
Description" / "with ASL" packs (Severance, Slow Horses ×5, Ted Lasso ×3, House of the
Dragon) + American Dad S22E08, a 2026 movie, a broken Laufey (missingFiles). All 0% /
un-imported. Removed them (+data) — both *arr queues were empty so no CDH/library risk
and no re-grab loop.

### The fix: re-grab from 1337x (public — no ratio, no slots)

1337x has all of R&M with hundreds of seeders, including x265 packs *smaller* than the
H.264 packs seedpool was choking on. Pulled per-season via Sonarr's release-search +
`POST /api/v3/release` (forced grab). Picks:

| S01 | S02 | S03 | S04 | S05 | S06 |
|-----|-----|-----|-----|-----|-----|
| 10.1 GB (DigitalCore BluRay x265 — see note) | 3.8 | 3.8 | 3.7 | 3.7 | 3.4 GB |

Gotchas hit while grabbing:
- **Flaky public indexer**: 1337x returns a *different* result set each query, so an
  exact-title match misses on some runs → needed ret/search + candidate fall-through.
- **HTTP 409 "Getting release from indexer failed"**: some specific releases have dead
  download links (the S03 UNSENSORED RARBG one, both seedpool S01 packs). Fall-through
  to the next candidate handled it.
- **S01 has no clean 1337x x265 season pack** (only "The Anime" — a different show — and
  a 3D-SBS rip). Grabbed S01 from **DigitalCore** instead (private, but NOT slot-locked
  — confirmed it downloaded fine). Hence S01 is 10 GB vs ~3.8 for the rest. Still a huge
  cut from 26.3 GB Remux.

### The other half: forcing the import (deliberate downgrade)

Sonarr **refuses to auto-import a downgrade** — every new x265 file was rejected
"Not an upgrade … Existing quality: Bluray-1080p Remux." So the downloads completed and
just sat there. Fix: drive Sonarr's **manual import API** (`GET /manualimport?folder=…`
then `POST /command {name:ManualImport, importMode:move}`) which overrides the rejection,
moves the x265 file into the library, and deletes the old Remux. Codified as
[`scripts/sonarr-force-import.py`](../scripts/sonarr-force-import.py).

Used **`move`** mode (not copy/hardlink): these are public-tracker grabs, ratio doesn't
matter, and move reclaims the space immediately. Recycle bin is off, so old Remux files
were deleted outright (no lingering copies). A background watcher imported each pack the
moment it finished and swept the dead public torrents after (left the 3 German-Remux
seeders alone).

### Result
- R&M: **163.2 → 37.1 GB**. drive4 now 72% (511 GB free).
- All seasons x265 1080p; library DB consistent; no orphans.
- seedpool packs + 13 junk torrents purged from qBit.

## Broken hardlinking — root-caused + migration plan written

Diagnosed the long-flagged "biggest structural storage win." Findings (verified live):
- **4 independent ext4 filesystems**, no pool (`/DATA` = drive1). Downloads pinned to
  `/mnt/drive1/torrents` (qBit `DefaultSavePath`); libraries on drive2/3/4.
- **`copyUsingHardlinks` is already True** in both *arr — so it's not a toggle bug. The
  apps try to hardlink and fall back to copy because `link()` can't cross filesystems
  (`EXDEV`). Pure **topology** problem.
- **Link-count audit: 0 hardlinked files** across all 6 roots (~3,900 files). drive1
  ones are nlink=1 only because their torrents already stopped seeding (expected).
- **~351 GB** sits in `torrents/complete`, most of it duplicated into cross-drive libs.

Fix = **mergerfs union pool** (already installed, `2.33.5` at `/usr/bin/mergerfs`) +
single `/data` tree so downloads and imported media co-locate on one disk and hardlink.
Full step-by-step migration, fstab persistence, reclaim strategy, and open decisions
written to [`docs/16-hardlinking-migration-plan.md`](../docs/16-hardlinking-migration-plan.md).
**Not executed** — needs an owner go + a low-use window (apps down during the in-place
`mv` reshape; no data moves between drives).
