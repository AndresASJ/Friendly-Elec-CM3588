# 2026-06-21

## Jellyseerr: fixed "You are offline" at home — missing NPM proxy host (split-DNS)

Owner reported Jellyseerr "wasn't working." Restarted the container first (clean start,
`Server ready on port 5055`, v2.7.3), but the real symptom turned out to be **"You are
offline" only on the home network** — it worked fine on cellular.

### Diagnosis
That split (works off-LAN, fails on-LAN) pointed straight at DNS/proxy, not the app:

- App healthy: `curl http://192.168.50.178:5055/api/v1/status` → **200**, and the raw
  `http://192.168.50.178:5055` loaded for the owner.
- **Cellular path:** `jellyseerr.asj.media` → public Cloudflare DNS → **Cloudflare Tunnel** →
  Jellyseerr. Works, bypasses the LAN entirely.
- **Home path:** AdGuard has a wildcard rewrite `*.asj.media → 192.168.50.178`
  (`/opt/adguardhome/conf/AdGuardHome.yaml`), so every `*.asj.media` name goes to **Nginx
  Proxy Manager** — but NPM had **no proxy host for jellyseerr**. Its hosts were jellyfin,
  photos, drive, adguard, tailscale, cloudflared, obsidian, homeassistant, writes, n8n, and
  the `asj.media` root — no seerr. So at home the request dead-ended at NPM and the Jellyseerr
  PWA showed "You are offline."

Root cause: the service had a Cloudflare Tunnel public hostname but was never added to NPM.

### Fix
Added an NPM proxy host for `jellyseerr.asj.media → http://192.168.50.178:5055`, mirroring the
working `jellyfin` entry (HTTP only on the LAN side — Cloudflare terminates TLS at the edge,
all NPM hosts here listen on `:80` only; websockets enabled; no SSL/Let's Encrypt needed).

NPM has no `sqlite3` binary (nor the host), and the API needs admin creds, so applied via
Python's `sqlite3` module directly against `/DATA/AppData/nginxproxymanager/data/database.sqlite`:

- Backed up the DB first → `database.sqlite.bak-20260621-001206`.
- Inserted `proxy_host` row **id 12** (domain `["jellyseerr.asj.media"]`, forward
  `192.168.50.178:5055`, `forward_scheme=http`, `allow_websocket_upgrade=1`, `enabled=1`,
  everything else mirroring jellyfin/id 4).
- Wrote the matching nginx conf `…/data/nginx/proxy_host/12.conf` (cloned from `4.conf`, port
  5055, server_name `jellyseerr.asj.media`, logs `proxy-host-12_*`).
- `docker exec nginxproxymanager nginx -t` → ok → `nginx -s reload`.

### Verified
- Through NPM with the real Host header:
  `curl -H "Host: jellyseerr.asj.media" http://192.168.50.178/api/v1/status` → **200**.
- AdGuard resolves `jellyseerr.asj.media → 192.168.50.178` (same as the working jellyfin),
  confirmed by a direct UDP DNS query to `192.168.50.178:53`.

Home path AdGuard → NPM → Jellyseerr is now complete. Owner needs a **hard refresh**
(`Ctrl+Shift+R`) to drop the stale "offline" service worker.

## Why
Jellyseerr was only ever wired up through the Cloudflare Tunnel, so it worked everywhere
*except* home, where AdGuard's wildcard forces `*.asj.media` through NPM. The general rule now
documented: every Cloudflare-Tunnel hostname needs a matching NPM proxy host (or an AdGuard
rewrite exception), or it works remotely but breaks on-LAN.

## Gotchas
- The "You are offline" banner is a **PWA/service-worker** message — it means *your browser*
  can't reach the backend, not that the server is down. Misleading; the backend was healthy
  the whole time.
- NPM (and the host) ship **no `sqlite3` CLI** — used Python's stdlib `sqlite3` instead.
- Manual DB edit is the unofficial path. It's consistent (DB row **and** nginx conf both
  written), so the host shows up correctly in the NPM UI; if it's ever edited + saved there,
  NPM regenerates `12.conf` from the same DB row. DB backup kept alongside the live file.
