# 2026-06-02

## Todo Capture: auto-file captures into Todoist sections

Enhanced the **Todo Capture → Todoist** n8n flow so captured tasks land in the right
**section** instead of piling up loose in Inbox-root. Previously Gemini's category was
only applied as a *label*; the task itself always went to Inbox with no section, so
everything texted to the ASJ Noti bot accumulated unsorted (this is what prompted the
big Todoist cleanup below).

### Change

Two edits to [`configs/n8n-workflow-todo-capture-todoist.example.json`](../configs/n8n-workflow-todo-capture-todoist.example.json):

- New **Get sections** node (`GET /api/v1/sections`, reuses the Todoist API
  credential) inserted between **Lookup contact** and **Build task**.
- **Build task** now matches Gemini's category (case-insensitive) against the fetched
  section names and sets `section_id` when one matches (Blog / Church / Coding / Kord /
  Routines). Updated the Gemini prompt in **Get message** to bias categories toward
  those real section names.

Design notes:
- **No hardcoded section IDs** — resolved at runtime, consistent with the flow's
  existing "no hardcoded project IDs" principle. Renaming/reordering sections won't
  break it.
- **Label only when unmatched** — if a category maps to a section, the task is *not*
  also tagged with a same-named label (the section/label redundancy is exactly what we
  just cleaned). Unmatched categories still fall back to a label.

Apply step (no n8n public API key configured): re-import the example in n8n → Workflows
→ Import from File, re-attach the Telegram / Gemini / Todoist / Postgres credentials
(the `CHANGE_ME` placeholders), and delete the old copy. Doc updated in
[`docs/15-n8n.md`](../docs/15-n8n.md).

## Hermes Agent installed (self-hosted AI agent on Telegram)

Set up [Hermes Agent](https://github.com/NousResearch/hermes-agent) (Nous Research) as a
new CasaOS app. Full writeup in [`docs/17-hermes.md`](../docs/17-hermes.md).

- **Build**: no published image — cloned the repo and built `hermes-agent:latest`
  locally on the CM5388 (aarch64, ~4.8 GB multi-stage s6-overlay build).
- **Brain**: Google **Gemini 3.5 Flash** (`google/gemini-3.5-flash`) via AI Studio key.
  Verified the key against the API — the older `gemini-2.0-flash` returned HTTP 429
  (`limit: 0` free tier), but `gemini-3.5-flash` returns 200, so pinned to that.
- **Gateway**: Telegram, long-polling (no port-forward). Locked to owner's user ID via
  `TELEGRAM_ALLOWED_USERS`. Secret redaction on by default.
- **Containers**: `hermes` (gateway) + `hermes-dashboard` (127.0.0.1:9119 only), both
  `network_mode: host`, data at `/mnt/drive1/AppData/hermes` → `/opt/data`.
- **Install**: registered with `casaos-cli app-management install -f compose/hermes.yml`
  so it shows in the CasaOS UI (not just raw compose). Confirmed `running`.
- Secrets in git-ignored `/mnt/drive1/AppData/hermes/hermes.env` (chmod 600); committed
  compose uses `env_file`, and [`configs/hermes.env.example`](../configs/hermes.env.example)
  documents the vars.

**⚠️ Open item — Telegram bot conflict.** Hermes was pointed at the existing
`@ASJNOTI_BOT` token. Telegram allows only one `getUpdates`/webhook consumer per token,
so if an n8n **Telegram-trigger** node uses the same bot, they'll fight over updates.
Recommend a dedicated `@BotFather` bot for Hermes; swap `TELEGRAM_BOT_TOKEN` in the env
file and `restart hermes`. (n8n send-only usage does not conflict.)

### Hermes follow-up: dedicated bot + model fix (same day)

Got Hermes fully working after the initial install:

- **Verified end-to-end**: `model=gemini-3.5-flash provider=gemini ... latency=5.3s`,
  bot replies on Telegram, allowlist confirmed (only owner ID `7419660817`).
- **Caveat**: Gemini free tier is **5 RPM** for gemini-3.5-flash; auxiliary calls (chat
  title generation) 429 while the main reply succeeds. Enable billing for heavier use.

Gotchas captured in [`docs/17-hermes.md`](../docs/17-hermes.md).

### Hermes round 2: single-container redesign + efficiency trim

Two real fixes after the bot was working:

1. **Single-container redesign (fixes s6-log lock).** The separate `hermes` +
   `hermes-dashboard` containers both mounted `/opt/data` and each spawned an s6-log for
   `gateways/default` → permanent `lock: Resource busy`, gateway never connecting. (Also
   saw an orphaned s6-log from a killed container holding the lock.) Collapsed to ONE
   container running the dashboard via `HERMES_DASHBOARD=1` under the same s6 tree.
   Reinstalled the CasaOS app from the new single-service `compose/hermes.yml`. Result:
   0 lock errors, Telegram connected, dashboard 127.0.0.1:9119 → HTTP 200.
2. **Efficiency / free-tier trim.** Free tier is 5 RPM **per model**. Routed all
   auxiliary tasks (vision, web_extract, session_search, title_generation, approval, mcp,
   compression) to `gemini-flash-lite-latest` via `provider: gemini` in `config.yaml` —
   separate quota bucket from the main `gemini-3.5-flash` replies, cheaper/faster, and no
   wasted OpenRouter/Nous fallback attempts.

Docs updated in [`docs/17-hermes.md`](../docs/17-hermes.md).
