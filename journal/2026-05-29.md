# 2026-05-29 — n8n: migrated SQLite → PostgreSQL

## Summary

Switched n8n's database backend from SQLite to PostgreSQL. Added a
`postgres:16-alpine` container (`n8n-postgres`) alongside n8n and pointed n8n at
it. Clean migration — the SQLite DB held no real data yet, so nothing was lost.

## Changes

- Added `n8n-postgres` (`postgres:16-alpine`) to the n8n CasaOS compose project,
  data at `/mnt/drive1/AppData/n8n-postgres/`, with a `pg_isready` healthcheck
- Pointed n8n at Postgres: `DB_TYPE=postgresdb`, `DB_POSTGRESDB_HOST=postgres`,
  user/db/password `n8n`; n8n `depends_on` Postgres `service_healthy`
- Moved both containers off the default `bridge` (`network_mode: bridge`) onto a
  user-defined network `n8n_default` so n8n resolves `postgres` by hostname
- Pinned `N8N_ENCRYPTION_KEY` explicitly in the compose (value unchanged, copied
  from the existing `config` file) so credentials stay decryptable
- Deployed via `docker compose up -d` in `/var/lib/casaos/apps/n8n/`
- Updated repo: `compose/n8n.yml` (sanitized, Postgres + network), `docs/15-n8n.md`

## Why

SQLite is fine at idle but locks under concurrent/scheduled workflow load.
Moving to Postgres now — while the instance is empty — avoids a painful
data migration later and gives a more robust backend for heavier automations.

## Gotchas

- **n8n does not auto-migrate SQLite → Postgres.** Flipping `DB_TYPE` just starts
  from an empty schema; the SQLite file is left untouched on disk but unused.
  Verified the SQLite DB first: 0 workflows / 0 credentials / 0 executions,
  only the initial owner-account user — so the cutover was safe.
- The owner account lived only in SQLite, so the first-run setup screen reappears
  once on Postgres — login must be re-created. (No workflows/creds to re-enter.)
- Pre-existing `n8n_default` network carried a stale `com.docker.compose.network`
  label (`default`), which blocked the first `up`. Fixed with
  `docker compose down` → `docker network rm n8n_default` → `up -d`.
- NPM is unaffected: port `5678` is still published, and NPM proxies via the host
  LAN IP (`192.168.50.178`), not the container network.

## Backups taken

Pre-migration snapshot at
`/mnt/drive1/AppData/n8n/_migration_backup_20260529-033935/`:
`docker-compose.yml.bak`, `database.sqlite.bak`, `config.bak`,
`postgres_password.txt`.

## Files touched

- `compose/n8n.yml` (Postgres service + user-defined network + encryption key)
- `docs/15-n8n.md` (stack table, data location, backup + migration notes)
- `journal/2026-05-29.md` (this file, new)

## What's next

- Add `/mnt/drive1/AppData/n8n-postgres/` (or a `pg_dump`) to `scripts/backup.sh`

# 2026-05-29 — n8n: first workflow (New Media → Telegram)

## Summary

Built and activated the first real n8n workflow: Sonarr/Radarr fire a webhook on
import, n8n formats a message and sends it to Telegram via the "ASJ Noti" bot
(@ASJNOTI_BOT).

## Changes

- Created workflow **"New Media → Notification"** (id `7o9QYSRy2zTWulS6`):
  Webhook `POST /webhook/media-added` → Code (Sonarr/Radarr payload detection +
  message formatting) → Telegram send
- Created an n8n **Telegram API** credential holding the bot token (stored in n8n
  / Postgres, never in the repo)
- Activated the workflow; verified end-to-end with synthetic Sonarr + Radarr
  payloads — both executions `success`, both Telegram messages delivered
- Exported a **sanitized** copy to
  `configs/n8n-workflow-new-media-notification.json` (token + chat ID replaced
  with `CHANGE_ME`)
- Documented setup + Sonarr/Radarr wiring in `docs/15-n8n.md`

## Why

User wanted phone notifications for new media. iMessage is impossible on Linux
(no Apple stack), and WhatsApp Cloud API is heavy (Meta business approval +
templates), so we used Telegram — free, ~2-min bot setup, lands on the phone.

## Gotchas

- Activating via the internal REST API needs the **dedicated route**:
  `POST /rest/workflows/<id>/activate` with a `versionId` in the body. Setting
  `active:true` via `PATCH /rest/workflows/<id>` returns 200 but silently leaves
  the workflow inactive.
- Telegram bots **cannot initiate** a chat — the user must message the bot once
  before `getUpdates` reveals the chat ID.
- For Sonarr/Radarr → n8n use the **host LAN IP** (`192.168.50.178:5678`), not a
  container hostname — they're on different Docker networks.

## Files touched

- `configs/n8n-workflow-new-media-notification.json` (new, sanitized export)
- `docs/15-n8n.md` (Workflows section)
- `journal/2026-05-29.md` (this entry)

# 2026-05-29 — n8n: three more workflows (torrent / disk / Soulseek)

## Summary

Added three Telegram-notifying workflows, all reusing the existing ASJ Noti bot:
**Torrent Done**, **Disk Guard**, and **Soulseek (slskd) Done**. All built via the
n8n REST API, activated, and tested end-to-end.

## Changes

- **A — Torrent Done → Telegram** (`/webhook/torrent-done`): qBittorrent's "run
  external program on completion" curls n8n; n8n formats name/category/size/files
  and flags uncategorized torrents. Tested with synthetic payloads.
- **D — Disk Guard → Telegram** (`/webhook/disk-report`): alerts when any `/mnt`
  drive crosses 85%, with state-change de-dup via workflow static data.
  - Host sensor `scripts/n8n-disk-report.sh` → `/usr/local/bin/` (read-only `df`)
  - Root cron `*/30 * * * *` installed (alongside the existing gluetun-qbit
    port-update cron)
  - Immediately flagged **drive2 at 100%** on first run.
- **E — Soulseek Done → Telegram**: 5-min poll of slskd
  `GET /api/v0/transfers/downloads`, groups newly-completed files by album, seeds
  history on first run. First scheduled execution: success.
  - Added a **private-network-scoped API key** to `slskd.yml`
    (`cidr: 192.168.0.0/16,172.16.0.0/12,10.0.0.0/8,127.0.0.1/32`) and restarted slskd
  - Stored the key in n8n as an HTTP Header Auth credential
- Sanitized exports for all three under `configs/*.example.json`
- Documented all three in `docs/15-n8n.md`

## Why

User wanted more workflows, specifically around torrent/download completion.
These cover the gaps the existing media-import alert doesn't: manual/uncategorized
torrents, disk pressure, and Soulseek music (which the *arr stack never touches).

## Gotchas

- **The n8n image is hardened.** No `n8n-nodes-base.executeCommand` ("Unrecognized
  node type"), and the Code-node sandbox blocks `require` (couldn't read disk via
  `fs` either). Rather than weaken the sandbox on a webhook-exposed service
  (`NODE_FUNCTION_ALLOW_BUILTIN=fs` was reverted), D gets disk stats from a host
  cron that POSTs to a webhook. n8n stays the brain; the host is just a sensor.
- Activating workflows via REST needs `POST /rest/workflows/<id>/activate` with a
  `versionId` body (the `PATCH active:true` route silently no-ops).
- slskd has **no** native completion hook → must poll its API (hence the API key).
- The n8n container has no `curl`; reachability tests inside it use `node` + `fetch`.

## Known issue surfaced (not n8n-related)

- **drive2 is 100% full (≈8 GB free).** Flagged for follow-up — writes to it will
  fail. `gluetun-qbit` and `slskd` were also showing unhealthy earlier.
