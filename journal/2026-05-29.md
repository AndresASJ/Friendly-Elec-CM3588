# 2026-05-29 ‚Äî n8n: migrated SQLite ‚Üí PostgreSQL

## Summary

Switched n8n's database backend from SQLite to PostgreSQL. Added a
`postgres:16-alpine` container (`n8n-postgres`) alongside n8n and pointed n8n at
it. Clean migration ‚Äî the SQLite DB held no real data yet, so nothing was lost.

## Changes

- Added `n8n-postgres` (`postgres:16-alpine`) to the n8n CasaOS compose project,
  data at `/mnt/drive1/AppData/n8n-postgres/`, with a `pg_isready` healthcheck
- Pointed n8n at Postgres: `DB_TYPE=postgresdb`, `DB_POSTGRESDB_HOST=postgres`,
  user/db/password `n8n`; n8n `depends_on` Postgres `service_healthy`
- Moved both containers off the default `bridge` (`network_mode: bridge`) onto a
  user-defined network `n8n_default` so n8n resolves `postgres` by hostname
- Pinned `N8N_ENCRYPTION_KEY` explicitly in the compose (value unchanged, copied
  from the existing `config` file) so credentials stay decryptable
- Deployed via `docker compose up -d` in `/var/lib/casaos/apps/n8n/`
- Updated repo: `compose/n8n.yml` (sanitized, Postgres + network), `docs/15-n8n.md`

## Why

SQLite is fine at idle but locks under concurrent/scheduled workflow load.
Moving to Postgres now ‚Äî while the instance is empty ‚Äî avoids a painful
data migration later and gives a more robust backend for heavier automations.

## Gotchas

- **n8n does not auto-migrate SQLite ‚Üí Postgres.** Flipping `DB_TYPE` just starts
  from an empty schema; the SQLite file is left untouched on disk but unused.
  Verified the SQLite DB first: 0 workflows / 0 credentials / 0 executions,
  only the initial owner-account user ‚Äî so the cutover was safe.
- The owner account lived only in SQLite, so the first-run setup screen reappears
  once on Postgres ‚Äî login must be re-created. (No workflows/creds to re-enter.)
- Pre-existing `n8n_default` network carried a stale `com.docker.compose.network`
  label (`default`), which blocked the first `up`. Fixed with
  `docker compose down` ‚Üí `docker network rm n8n_default` ‚Üí `up -d`.
- NPM is unaffected: port `5678` is still published, and NPM proxies via the host
  LAN IP (`192.168.50.178`), not the container network.

# 2026-05-29 ‚Äî n8n: first workflow (New Media ‚Üí Telegram)

## Summary

Built and activated the first real n8n workflow: Sonarr/Radarr fire a webhook on
import, n8n formats a message and sends it to Telegram via the "ASJ Noti" bot
(@ASJNOTI_BOT).

## Changes

- Created workflow **"New Media ‚Üí Notification"** (id `7o9QYSRy2zTWulS6`):
  Webhook `POST /webhook/media-added` ‚Üí Code (Sonarr/Radarr payload detection +
  message formatting) ‚Üí Telegram send
- Created an n8n **Telegram API** credential holding the bot token (stored in n8n
  / Postgres, never in the repo)
- Activated the workflow; verified end-to-end with synthetic Sonarr + Radarr
  payloads ‚Äî both executions `success`, both Telegram messages delivered
- Exported a **sanitized** copy to
  `configs/n8n-workflow-new-media-notification.json` (token + chat ID replaced
  with `CHANGE_ME`)
- Documented setup + Sonarr/Radarr wiring in `docs/15-n8n.md`

## Why

User wanted phone notifications for new media. iMessage is impossible on Linux
(no Apple stack), and WhatsApp Cloud API is heavy (Meta business approval +
templates), so we used Telegram ‚Äî free, ~2-min bot setup, lands on the phone.

## Gotchas

- Activating via the internal REST API needs the **dedicated route**:
  `POST /rest/workflows/<id>/activate` with a `versionId` in the body. Setting
  `active:true` via `PATCH /rest/workflows/<id>` returns 200 but silently leaves
  the workflow inactive.
- Telegram bots **cannot initiate** a chat ‚Äî the user must message the bot once
  before `getUpdates` reveals the chat ID.
- For Sonarr/Radarr ‚Üí n8n use the **host LAN IP** (`192.168.50.178:5678`), not a
  container hostname ‚Äî they're on different Docker networks.

## Files touched

- `configs/n8n-workflow-new-media-notification.json` (new, sanitized export)
- `docs/15-n8n.md` (Workflows section)
- `journal/2026-05-29.md` (this entry)
